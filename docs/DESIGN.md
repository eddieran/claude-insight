## Problem Statement

Claude Code is the most capable local AI coding agent available, but it's a black box at runtime. When something goes wrong — a surprising file edit, an unexpected tool call, a permission escalation, a runaway bash command — there's no clean way to answer: what happened, what triggered it, what context was available, what policy allowed it, and what changed in the world.

Existing Claude Code observability tools (disler/claude-code-hooks-multi-agent-observability, simple10/agents-observe, ColeMurray/claude-code-otel) all do the same thing: capture hook events, show a dashboard. None correlate hooks with transcripts. None model instruction provenance or permission decision chains. None offer session replay. None have full-text search across sessions.

Generic LLM observability platforms (Langfuse, LangSmith, Braintrust) are cloud-hosted, not Claude-Code-specific, and don't understand hooks, CLAUDE.md instruction loading, sandbox boundaries, MCP server trust maps, or the 27 lifecycle events Claude Code exposes.

The gap: no tool builds a forensic evidence chain across Claude Code's multiple observable surfaces into a unified, queryable, replayable local evidence graph.

## What Makes This Cool

Time-travel debugging for AI agent sessions. Scrub through a Claude Code session like a video: see every instruction load, every tool call, every permission decision, every context window state change, every file mutation, in order, with full causal chain linking. Click any event and see everything that led to it and everything that followed.

This is Chrome DevTools for AI coding agents.

## Constraints

- Must be local-first (no server, no account, no cloud dependency for V1)
- Must not depend on hidden chain-of-thought or unpublished internals
- Must handle Claude Code version drift (raw-first storage, version-tagged records)
- Must separate metadata from sensitive content (privacy by default)
- Must not degrade Claude Code performance (async audit hooks, bounded timeouts)
- Source code research was done against Claude Code source at ~/Projects/claude_code

## Premises

1. Claude Code's observable surfaces (27 hook events, JSONL transcripts, stream-json, OTel, status line, config files) are stable enough to build a product on.
2. Multi-source correlation (joining hooks + transcripts + stream-json by session_id, tool_use_id, promptId) is the key differentiator vs. existing tools.
3. Local-first SQLite + FTS is the right storage for V1.
4. Session replay is the entry point experience that unlocks audit, permission, instruction, and tool lineage views.
5. Cross-session intelligence (pattern detection, friction signals, risk signals) is V2, not V1.

## Cross-Model Perspective

Codex cold read highlights:

- **Coolest version not yet considered:** A forkable counterfactual debugger. Pause at any decision point, branch with one changed variable (different instruction, permission, sandbox mode, tool result), show divergence. Turns observability into a behavioral lab. (V2/V3 candidate.)
- **Key insight:** "the agent that learns about the agent" — replay is the product hook, but cross-session intelligence is the obsession.
- **50% existing tool:** OTel Collector + Jaeger gives ingestion pipeline architecture and span/trace model. The 50% to build: Claude-specific receivers, canonical session schema, multi-source correlation, instruction provenance, permission chains, local SQLite, replay UI.
- **Weekend build:** Single-session replay from transcript + hooks into SQLite. One events table, correlator, three-pane web UI. Click any event, highlight the full causal chain.

## Approaches Considered

### Approach A: Evidence Recorder + CLI (Minimal Viable)
Hook capture + transcript ingestion + SQLite + CLI replay. No web UI. Ships fastest but the "whoa" moment is weaker. Terminal-only limits security/compliance personas. Completeness: 6/10.

### Approach B: Full Local Observatory (Chosen)
Hook capture + transcript + stream-json + config snapshots into SQLite with FTS. Local web UI with session replay, instruction provenance, permission audit, tool lineage. Three-pane replay (timeline, transcript, evidence panel). Feature coverage: 9/10. Implementation readiness: 7/10 (schema, correlation, and UI need detailed design, addressed below).

### Approach C: Perfetto-First Demo (Creative/Lateral)
Export sessions as enhanced Perfetto traces via existing CLAUDE_CODE_PERFETTO_TRACE. Build Claude-specific viewer. Immediate visual payoff but locks into Perfetto's model. Completeness: 5/10.

## Recommended Approach

**Approach B: Full Local Observatory**

Multi-source correlation + web replay is what makes this genuinely different from everything else that exists. The design doc already scopes it well. The data model is the product — the UI is how you prove it works.

**Security/compliance scope for V1:** V1 includes permission audit view (which tools were allowed/denied, by what rule) and instruction provenance view (which CLAUDE.md files were loaded). These serve security personas without requiring tamper-evident storage or formal compliance export formats, which are V2. The append-only raw JSONL store provides basic audit trail integrity.

**Future vision (V2/V3):** Counterfactual debugger (fork sessions at decision points, branch with changed variables, show divergence). Cross-session intelligence (friction signals, risk patterns, anomaly detection). These are NOT part of V1 schema design — the V1 schema must be correct for V1 use cases, not preemptively designed for speculative features.

## Design Decisions (Resolved)

### D1: Tech stack — Rust + Ratatui TUI (REVISED from React/Vite)
**Language:** Rust. **TUI:** Ratatui + crossterm. **HTTP:** axum (daemon for hook events). **Storage:** rusqlite. **JSON:** serde_json + serde. **CLI:** clap. **Distribution:** Single binary via `cargo install --git https://github.com/eddieran/claude-insight --bin claude-insight` + GitHub Releases (linux/darwin/windows, amd64/arm64). No npm, no node_modules, no native compilation issues. Just download and run.

Rationale: The user chose TUI over web UI. For a complex three-pane terminal debugger with syntax highlighting, scrollable lists, and animations, Ratatui is the gold standard. Single-binary distribution eliminates all packaging friction.

### D2: Hook installation — `claude-insight init` auto-installs
`claude-insight init` writes hook entries into `.claude/settings.json` (project-level) or `~/.claude/settings.json` (global). All hooks use the `http` hook type to POST to the axum daemon. JSONL fallback if daemon is down. SessionStart hook auto-launches daemon if not running. Max 50ms per hook for subsequent events (first hook cold-starts daemon, which is slower and acceptable).

### D2a: Architecture — Daemon with separate capture port (from eng review)
Capture (hook HTTP receiver) runs on a dedicated port. TUI connects to daemon via local API on a separate port. If TUI crashes, capture keeps running. Codex review identified this as a failure isolation requirement.

### D2b: Storage model — Raw-first, normalize on demand (from eng review)
Write to raw_events only on ingest. Normalized tables (sessions, tool_invocations, etc.) are materialized on demand or via `claude-insight normalize` command. This allows correlation logic to be improved without re-capturing data. Codex review identified this as critical for a system where correlation heuristics (timestamp proximity) will evolve.

### D2c: JSONL fallback (from eng review)
If daemon is down when a hook fires, the hook script falls back to appending to `~/.claude-insight/backlog.jsonl`. Daemon processes backlog on startup. Events never lost.

### D3: Subagent correlation — parent-child event linking
Subagent transcripts at `{sessionId}/subagents/agent-{agentId}.jsonl` are ingested as child traces. Linked to parent session via `agent_id` field present in hook payloads and transcript entries. Displayed in the timeline as expandable sub-traces (collapsed by default, expandable on click).

### D4: Cross-session boundary — search is V1, intelligence is V2
Cross-session FTS *search* (find all sessions that used tool X, touched file Y, loaded instruction Z) is V1. Cross-session *pattern detection* (friction signals, risk signals, anomaly analysis) is V2.

### D5: Content capture — metadata-first, configurable via CLI
Default: metadata + content references (file paths, tool names, hashes). Full content capture enabled via `claude-insight init --capture-content` or `capture_content: true` in config. Redaction of common secret patterns applied automatically.

## Data Model (Draft Schema)

### Evidence Graph = Relational Tables with Foreign Keys

The "evidence graph" is a set of normalized SQLite tables with foreign key relationships and FTS virtual tables. Not a graph database.

### Core Tables

```sql
-- Sessions
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,           -- session_id (UUID)
  transcript_path TEXT,
  cwd TEXT,
  project_dir TEXT,
  claude_version TEXT,
  model TEXT,
  permission_mode TEXT,
  start_ts TEXT,                 -- ISO timestamp
  end_ts TEXT,
  end_reason TEXT,
  source TEXT,                   -- 'startup'|'resume'|'clear'|'compact'
  cost_usd REAL,
  input_tokens INTEGER,
  output_tokens INTEGER,
  raw_artifact_path TEXT         -- path to raw JSONL backup
);

-- Prompts (user submissions)
CREATE TABLE prompts (
  id TEXT PRIMARY KEY,           -- promptId (UUID)
  session_id TEXT REFERENCES sessions(id),
  prompt_text TEXT,              -- null if metadata-only mode
  prompt_hash TEXT,              -- SHA-256 of prompt text
  ts TEXT,
  -- instruction context derived from instruction_loads WHERE session_id = this.session_id AND ts <= this.ts
);

-- Tool Invocations
CREATE TABLE tool_invocations (
  id TEXT PRIMARY KEY,           -- tool_use_id
  session_id TEXT REFERENCES sessions(id),
  prompt_id TEXT REFERENCES prompts(id),
  tool_name TEXT NOT NULL,
  tool_input_json TEXT,          -- null if metadata-only
  tool_input_hash TEXT,
  tool_response_json TEXT,       -- null if metadata-only
  tool_response_hash TEXT,
  is_mcp BOOLEAN DEFAULT 0,
  mcp_server_name TEXT,
  agent_id TEXT,                 -- null for main agent
  pre_hook_ts TEXT,
  post_hook_ts TEXT,
  duration_ms INTEGER,
  success BOOLEAN,
  error_text TEXT
);

-- Permission Decisions
CREATE TABLE permission_decisions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT REFERENCES sessions(id),
  tool_invocation_id TEXT REFERENCES tool_invocations(id),
  decision TEXT NOT NULL,        -- 'allow'|'deny'|'ask'
  source TEXT,                   -- 'rule'|'mode'|'classifier'|'hook'|'user'
  rule_text TEXT,                -- matching rule if identifiable
  permission_mode TEXT,
  ts TEXT
);

-- Instruction Loads
CREATE TABLE instruction_loads (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT REFERENCES sessions(id),
  file_path TEXT NOT NULL,
  memory_type TEXT NOT NULL,     -- 'User'|'Project'|'Local'|'Managed'
  load_reason TEXT NOT NULL,     -- 'session_start'|'nested_traversal'|'path_glob_match'|'include'|'compact'
  trigger_file_path TEXT,
  parent_file_path TEXT,
  content_hash TEXT,
  ts TEXT
);

-- Config Snapshots
CREATE TABLE config_snapshots (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT REFERENCES sessions(id),
  file_path TEXT NOT NULL,
  file_hash TEXT NOT NULL,
  scope TEXT,                    -- 'user'|'project'|'local'|'managed'
  content_json TEXT,             -- null if metadata-only
  ts TEXT
);

-- Raw Events (append-only, all sources)
CREATE TABLE raw_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT,
  source TEXT NOT NULL,          -- 'hook'|'transcript'|'stream_json'|'config'
  event_type TEXT NOT NULL,
  ts TEXT NOT NULL,
  tool_use_id TEXT,              -- nullable join key
  prompt_id TEXT,                -- nullable join key
  agent_id TEXT,
  payload_json TEXT,
  claude_version TEXT,
  adapter_version TEXT
);

-- FTS index (contentless — requires explicit sync)
CREATE VIRTUAL TABLE events_fts USING fts5(
  session_id, event_type, tool_name, file_path, prompt_text, content,
  content=raw_events, content_rowid=id
);

-- Sync trigger: auto-populate FTS on raw_events insert
CREATE TRIGGER raw_events_ai AFTER INSERT ON raw_events BEGIN
  INSERT INTO events_fts(rowid, session_id, event_type, tool_name, file_path, prompt_text, content)
  VALUES (new.id, new.session_id, new.event_type,
          json_extract(new.payload_json, '$.tool_name'),
          json_extract(new.payload_json, '$.file_path'),
          json_extract(new.payload_json, '$.prompt'),
          new.payload_json);
END;
```

## Failure Modes

| Failure | Impact | Mitigation |
|---------|--------|------------|
| Malformed transcript JSONL | Unparseable lines skipped | Log warning, continue ingestion. Raw file preserved for manual inspection. |
| Hook timeout (>50ms) | Hook output dropped | Hooks use `async: true`. JSONL append is <1ms. If append fails, event lost (non-critical). |
| SQLite FTS indexing failure | Search degraded | FTS rebuild from raw_events table. `claude-insight rebuild-index` command. |
| Claude Code version drift | Payload schema changes | Raw-first storage preserves original payloads. Version tag on every record. Normalization errors logged, raw events still queryable. |
| Disk space exhaustion | Capture stops | Default retention: 90 days metadata, 30 days content. `claude-insight gc` command prunes old sessions. Max disk budget configurable. |
| Concurrent write conflicts | Corrupted JSONL | Single-writer pattern: one recorder process per session. Append-only with file locking. |

## Retention Policy

- Metadata (raw_events, permission_decisions, instruction_loads): 90 days default
- Full content (prompt text, tool I/O): 30 days default
- Raw JSONL artifacts: 90 days default
- Config snapshots: 90 days default
- `claude-insight gc` prunes expired records
- `claude-insight gc --aggressive` removes content but keeps metadata
- Configurable via `~/.claude-insight/config.json`
